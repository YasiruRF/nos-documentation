Capabilities & Limits of NOS

While NOS accelerates development by automating boilerplate, it is designed as a focused abstraction layer. Understanding what NOS can and cannot do will help you determine when to leverage the DSL and when to drop down to raw ROS2 APIs.

What NOS Can Do (Capabilities)

  • Declarative Node Configuration: Declare topics, publishers, subscribers, parameters, services, and actions in a clean, high-level interface.
  • Automatic ROS2 Setup: Generates imports, node class inheritance, parameter registrations, and property getters (e.g. self.my_parameter).
  • Parameter Bounds Checking: Enforce runtime parameter rules (e.g. @range and @one_of) by automatically mapping them to official ROS2 ParameterDescriptor limits.
  • QoS Mapping: Customize reliability, durability, history, and queue depth inline using the @qos helper annotation.
  • Managed Node Lifecycles: Declare lifecycle: managed to inherit a deterministic state-transition machine, automatically routing configuration and activation hooks to your methods.
  • Action Server Code Generation: Creates fully compliant Action Servers, mapping user-implemented goal execution callbacks directly to the ROS2 Action Server wrapper.
  • Reentrant Callback Groups: Attach the @reentrant annotation to any subscription, service, or action to automatically configure a ReentrantCallbackGroup, preventing runtime deadlocks.
  • Service & Action Clients: Attach the @client annotation to any service or action block entry to compile it as an outgoing ROS2 client (create_client or ActionClient).
  • Inline ROS2 Interface Creation: Define custom messages, services, and actions directly inside your `.nos` files (e.g. message Point @id(10) { x: float }). NOS will compile these into official `.msg`, `.srv`, and `.action` package files.
  • Launch System Descriptions: Coordinate namespaces, dynamic arguments, composable component containers, and nested configurations in the launch DSL to generate `*.launch.py` scripts.
  • Security Sanitization: Inspects all injected Python code blocks at compile time using AST (Abstract Syntax Tree) walking to prevent command injections and warn on file/network access.

What NOS Cannot Do (Limits & Boundaries)

1. No Native C++ Node Code Generation

NOS only generates Python-based ROS2 packages (ament_python topology). The compiler does not transpile your declarative model or injected code blocks into C++ (rclcpp) packages.

2. No Multi-threaded Executor Control

NOS compiles packages using standard single-threaded executors. If your nodes require complex, multi-threaded thread pool execution or custom event loop executors, you must manage these executors manually in python scripts.

3. Limited Nested Parameter Types

NOS parameters only support primitive types (int, float, double, string, bool), lists (list<type>), and simple flat custom structures (struct). It does not support complex nested maps or dictionaries.
