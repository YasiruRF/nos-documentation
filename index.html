Introduction to NOS

Node Orchestration System (NOS) is a declarative Domain-Specific Language (DSL) and compiler designed to streamline ROS2 robotics development. By describing your robot's architecture declaratively, NOS eliminates the boilerplate, file fragmentation, and type fragility typically associated with standard ROS2 programming.

Conceptual Mapping: NOS vs. Official ROS2

NOS is built on top of official ROS2 standards. The table below outlines how the declarative concepts in NOS map directly to their runtime behaviors in standard ROS2:

NOS Concept ROS2 Design Equivalent Official ROS2 Behavior
node block ROS2 Node Compiles to a class inheriting from rclpy.node.Node or LifecycleNode, representing a single participant in the ROS graph.
parameters block ROS2 Parameters Maps to declared node parameters, registering default values and generating type descriptors with range limits.
subscriptions block ROS2 Subscribers Registers node subscriptions, matching specific message structures and executing safety wrapper callbacks on incoming data.
publications block ROS2 Publishers Creates publisher instances configured with a specific topic and dynamic Quality of Service profiles.
services block ROS2 Services Maps to a Service Server, routing client requests to synchronous Python methods inside the node class.
actions block ROS2 Actions Registers Action Servers, enabling long-running, feedback-rich asynchronous goal processing.
lifecycle: managed Managed Nodes Compiles into a state-transition lifecycle node, routing transition states (Unconfigured, Active, etc.) to user methods.
launch block ROS2 Launch Descriptions Generates Python launch modules (`*.launch.py`) to manage execution, namespaces, parameter values, and component containers.
