Nodes & Layout Syntax

Package Decl

Every NOS file must start with a package statement. Packages correspond to ROS2 package names and encapsulate all declared nodes and launch layouts. You can also declare external package dependencies (e.g. standard message or service interfaces) to be registered automatically in the generated build configurations (such as package.xml): 

package package_name
version "1.0.0"
depends: ["std_msgs", "std_srvs"]

Nodes

A node represents a standard executable class. NOS transpiles this into a subclass of rclpy.node.Node. 

node NodeName {
    # Blocks inside
}

Parameters & Validation

The parameters block maps to ROS2 Node Parameters. You declare parameter names, types, default values, and optional range validation annotations. 

parameters {
    # Default types
    name: string = "robot_one"
    max_linear_speed: float = 0.5 @range(0.0, 2.0)
    update_rate: int = 10 @range(1, 100)
    mode: int = 1 @one_of(1, 2, 3)
}
🔗 ROS2 Parameters Mapping

NOS compiles these into official ROS2 ParameterDescriptor objects. The ranges are registered internally using FloatingPointRange or IntegerRange, which automatically enables ROS2 dynamic parameter safety validation.

Publications & Subscriptions

Topics represent unidirectional, many-to-many communication channels. They are used for continuous high-frequency data streams, such as sensor readings or velocity commands.

DSL Syntax

subscriptions {
    scan: sensor_msgs::LaserScan @topic("/scan")
}

publications {
    cmd_vel: geometry_msgs::Twist @topic("/cmd_vel") @qos(reliable, depth=10)
}
🔗 Under the Hood: Topics Implementation
  • Subscribers: The compiler generates a call to self.create_subscription() inside _setup_communications(). It automatically registers a hidden, private wrapper callback _on_[name]_received(self, msg). This wrapper catches any user-code execution exceptions and logs them via self.get_logger().error(), protecting the node process from fatal crashes.
  • Publishers: Spawns a self.create_publisher() call. The compiler exposes the publisher as a public class attribute self.[name], allowing you to call self.[name].publish(msg) from any execution handler.

QoS (Quality of Service) Configurations

The @qos annotation maps directly to ROS2 Quality of Service Profiles. Supported arguments include:

  • Reliability: reliable, best_effort (Corresponds to ReliabilityPolicy).
  • Durability: transient_local, volatile (Corresponds to DurabilityPolicy).
  • History: keep_last, keep_all (Corresponds to HistoryPolicy).
  • Depth: depth=integer_value (Corresponds to Queue size).

Action Servers (Actions)

Actions are designed for long-running, asynchronous tasks that take time to complete (such as navigating robot arms or moving to a coordinate). Actions provide goal control, feedback updates, and final result reporting.

DSL Syntax

# 1. Declare the action interface
action NavigateToGoal @id(40) {
    goal {
        target_pose: geometry_msgs::PoseStamped
    }
    feedback {
        current_pose: geometry_msgs::PoseStamped
        distance_remaining: float
    }
    result {
        success: bool
        total_time: float
    }
}

node NavigationNode {
    # 2. Register the action server
    actions {
        navigate: NavigateToGoal
    }
}
🔗 Under the Hood: Action Servers Implementation
  • Action Server Initialization: The compiler instantiates a fully compliant ROS2 ActionServer inside the _setup_communications() initialization hook, passing your node, action message class, action name, and a goal execution callback.
  • Execute Wrappers: The compiler generates a private wrapper method _on_navigate_execute(self, goal_handle). If the user-defined callback fails or throws an exception, the wrapper intercepts the crash, logs the traceback, and automatically calls goal_handle.abort() to maintain graph stability.
  • Goal Handling: Developers write their execution loop in the on_navigate_execute(self, goal_handle) callback. Periodic updates are dispatched by calling goal_handle.publish_feedback(feedback_msg). Once complete, you call goal_handle.succeed() and return a compiled Action Result object.
  • Goal & Cancel Validation: If you declare callback handlers named on_[action_name]_goal(self, goal_request) or on_[action_name]_cancel(self, goal_handle), the compiler registers them as the goal_callback and cancel_callback on the ActionServer. Returning GoalResponse.ACCEPT (or REJECT) and CancelResponse.ACCEPT (or REJECT) allows validating goal arguments before task execution begins.

Concurrency & Clients (Callback Groups & Outgoing Calls)

Robotic nodes frequently need to initiate outgoing requests (clients) or handle concurrent execution loops safely without blocking callback execution threads (causing deadlock).

DSL Syntax

node SensorGateway {
    subscriptions {
        # Trigger reentrant threading for this subscription
        sensor_data: sensor_msgs::LaserScan @topic("/scan") @reentrant
    }

    services {
        # Declare an outgoing service client (using @client annotation)
        add_service: example_interfaces::AddTwoInts @service("/add") @client
    }

    actions {
        # Declare an outgoing action client
        nav_client: nav2_msgs::NavigateToPose @client @reentrant
    }
}
🔗 Under the Hood: Concurrency & Clients Implementation
  • Reentrant Callback Groups (@reentrant): The compiler automatically instantiates a ROS2 ReentrantCallbackGroup inside the node class constructor. It registers the subscription, publisher, service, or action to use this callback group, enabling multithreaded concurrent callback executions.
  • Outgoing Service Clients (@client inside services): Compiles to a ROS2 Service Client self.add_service = self.create_client(AddTwoInts, "/add"). No server-side request callback wrappers are generated.
  • Outgoing Action Clients (@client inside actions): Compiles to a ROS2 Action Client self.nav_client = ActionClient(self, NavigateToPose, 'nav_client').
