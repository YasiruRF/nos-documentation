Cookbook & Task Reference

This section is a task-based reference. Find the task you want to perform and copy the declarative template code.

Task 1: If you want to declare parameters and enforce range checks

Declare type assignments and default values within the parameters block. Attach @range(min, max) for floating or integer limits, or @one_of(val1, val2...) for state machine presets. 

node SensorConfig {
    parameters {
        # Float variable bound to range
        frequency: float = 10.0 @range(1.0, 50.0)
        
        # Integer state variable
        buffer_depth: int = 10 @range(1, 100)
        
        # Choice selection
        operational_mode: int = 1 @one_of(1, 2, 3)
    }
}

Task 2: If you want to declare custom interfaces (inline msg, srv, or actions)

Rather than creating external folders containing `.msg`, `.srv`, or `.action` files, you can declare your interfaces directly inside the `.nos` file at the root level.

A. Custom message type

# ID acts as a compiler identification tag
message FilteredScan @id(10) {
    min_angle: float
    max_angle: float
    ranges: list
    intensities: list
}

B. Custom service type

service TriggerCalibrate @id(20) {
    request {
        sample_size: int
    }
    response {
        success: bool
        offsets: list
    }
}

C. Custom action type

action RotateRobot @id(30) {
    goal {
        angle: float
    }
    feedback {
        current_angle: float
    }
    result {
        total_rotation: float
        completed: bool
    }
}

Task 3: If you want to configure custom Quality of Service (QoS)

Append the @qos annotation to your publication or subscription declaration. 

publications {
    # Reliable, keep_last history with depth limit
    laser_out: sensor_msgs::LaserScan @topic("/laser") @qos(reliable, depth=10)
    
    # Best Effort, transient local durability (good for map or state static publishers)
    map_out: nav_msgs::OccupancyGrid @topic("/map") @qos(best_effort, transient_local, depth=1)
}

Task 4: If you want to orchestrate composable nodes in containers

In launch layouts, declare a container target, name the process, and list the composable component plugin nodes to be run within the single process executor. 

launch MultiNodeLauncher {
    container sensors_container @name("sensor_ipc_container") {
        lidar: PointCloudFilter
        camera: ImageProcessor
    }
}

Task 5: If you want to remap topics dynamically at launch

To change topics without editing node code, declare topic remapping variables inside the node instance config block in the launch DSL: 

launch NavigationLaunch {
    group navigation_nodes @namespace("robot") {
        controller: NavigationController {
            REMAP: {
                "/cmd_vel" -> "/cmd_vel_muxed",
                "/odom" -> "/localization/odom"
            }
        }
    }
}

Task 6: If you want to declare a Lifecycle Manager in your launch configuration

Manage state transitions for a group of managed nodes deterministically using the lifecycle manager declaration in the launch DSL: 

launch VehicleBringup {
    # Declaring the state transition manager
    lifecycle manager AutoManager {
        manages: [camera_driver, controller_node]
        transitions: [
            { from: unconfigured, to: inactive },
            { from: inactive, to: active }
        ]
    }
}
